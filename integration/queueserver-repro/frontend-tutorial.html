IOS queueserver — frontend quick start

Run the real bluesky-queueserver + bluesky-httpserver against the NSLS-II IOS profile collection, backed by simulated EPICS IOCs, and drive it — including the beamline E_ramp fly scan — over HTTP.

1. Prerequisites

  • Docker (runs redis, mongodb, kafka)
  • git, curl, openssl
  • pixi — auto-installed to ~/.pixi if missing

2. Run it

git clone https://github.com/NSLS2/ophyd-service.git
cd ophyd-service
git checkout simple-queueserver-ios-iocs
./integration/queueserver-repro/reproduce.sh up

First run is slow (clones the profile and solves its pixi environment — several minutes). When it finishes it prints 145 allowed plans / 100 allowed devices, the HTTP API URL, and a generated API key, and leaves everything running.

Handy commands: reproduce.sh status, reproduce.sh logs, reproduce.sh down (stop), reproduce.sh nuke (stop + delete work dir).

Unlike a fixed-key demo, the API key is generated per install and saved to $HOME/qs-repro/config/secrets.env. Read it with: 
KEY=$(sed -n 's/^HTTP_API_KEY=//p' "$HOME/qs-repro/config/secrets.env")

3. Simulated devices

Six purpose-built IOCs give realistic behavior for the mono energy, current amplifiers, EPU, vortex MCA, scaler, and feedback loop; a catch-all "blackhole" IOC answers every other PV so the whole profile connects. That means plans that read the scaler (sclr) or ramp the mono energy (pgm/pgm_energy) produce moving, non-trivial values.

4. Connect from React

Two values are all the UI needs — put them in .env (use the generated key from step 2):

VITE_QSERVER_URL=http://localhost:60610
VITE_QSERVER_API_KEY=<paste HTTP_API_KEY from secrets.env>
Dev vs. production URL. This dev stack exposes the httpserver directly on the host at http://localhost:60610 (no TLS, no proxy). In a real NSLS-II deployment the httpserver binds localhost:60610 and is not reachable off-box; an nginx reverse proxy terminates TLS on 443 and forwards to it (port 80 redirects to 443). The only thing the frontend changes between environments is the base URL:
Base URL the UI usesWhat serves it
dev (this stack)http://localhost:60610httpserver, exposed directly on the host port
productionhttps://<qs-host>/nginx :443 (TLS) → proxy_pass → localhost:60610 httpserver
The API paths, the Authorization: ApiKey … header, and the request/response bodies are identical in both — nginx only adds TLS and the standard Upgrade/Connection headers so the live console stream keeps working. Set VITE_QSERVER_URL to https://<qs-host>/ for production; everything else in this guide is unchanged.
const QS = import.meta.env.VITE_QSERVER_URL;
const KEY = import.meta.env.VITE_QSERVER_API_KEY;
const headers = { Authorization: `ApiKey ${KEY}`, "Content-Type": "application/json" };

// 1. open the environment (loads the IOS profile) — do this once
await fetch(`${QS}/api/environment/open`, { method: "POST", headers });

// 2. poll until the worker is ready
async function waitReady() {
  for (;;) {
    const s = await fetch(`${QS}/api/status`, { headers }).then(r => r.json());
    if (s.worker_environment_exists) return;
    await new Promise(r => setTimeout(r, 2000));
  }
}
await waitReady();
Auth: /api/ping and /api/status work without a key; everything else needs it. Simplest: send it on all requests.

5. Ready-to-run examples

Each plan can be queued (queue/item/add then queue/start) or run immediately (queue/item/execute). The examples below use immediate execute. Poll /api/status until manager_state is idle, then read /api/history/get.

a. Count the scaler (quick sanity check)

# curl — reads the (realistic) scaler 3 times
curl -s -X POST "$QS/api/queue/item/execute" \
  -H "Authorization: ApiKey $KEY" -H "Content-Type: application/json" \
  -d '{"item":{"name":"count","args":[["sclr"]],"kwargs":{"num":3,"delay":0.2},"item_type":"plan"}}'

b. E_ramp — the mono-energy fly scan

Signature: E_ramp(dets, start_eV, stop_eV, velocity_eV_per_s). It sets the mono fly start/stop/velocity, starts the fly, and reads dets repeatedly while the energy ramps, finishing when the fly reports Ready.

# curl — ramp 850 → 852 eV at 0.5 eV/s, reading the scaler (~4 s)
curl -s -X POST "$QS/api/queue/item/execute" \
  -H "Authorization: ApiKey $KEY" -H "Content-Type: application/json" \
  -d '{"item":{"name":"E_ramp","args":[["sclr"],850,852,0.5],"item_type":"plan"}}'

# a longer, slower ramp: 850 → 860 eV at 1.0 eV/s (~10 s)
curl -s -X POST "$QS/api/queue/item/execute" \
  -H "Authorization: ApiKey $KEY" -H "Content-Type: application/json" \
  -d '{"item":{"name":"E_ramp","args":[["sclr"],850,860,1.0],"item_type":"plan"}}'

# ramp back down: 860 → 850 eV at 2.0 eV/s (~5 s)
curl -s -X POST "$QS/api/queue/item/execute" \
  -H "Authorization: ApiKey $KEY" -H "Content-Type: application/json" \
  -d '{"item":{"name":"E_ramp","args":[["sclr"],860,850,2.0],"item_type":"plan"}}'
// JS fetch — queue E_ramp and start it
async function runERamp(dets, startEV, stopEV, velocity) {
  await fetch(`${QS}/api/queue/item/add`, { method: "POST", headers,
    body: JSON.stringify({ item: {
      name: "E_ramp",
      args: [dets, startEV, stopEV, velocity],
      item_type: "plan",
    }})});
  await fetch(`${QS}/api/queue/start`, { method: "POST", headers });
}
await runERamp(["sclr"], 850, 852, 0.5);
Args: dets is a list of device names (any from /api/devices/allowed; sclr is a good default). start_eV/stop_eV are mono energies (valid range ~200–2200 eV), and velocity is the ramp rate in eV/s — the ramp takes roughly |stop − start| / velocity seconds.

c. Watch it happen

# poll status until idle
curl -s "$QS/api/status" -H "Authorization: ApiKey $KEY"
# then inspect the completed run
curl -s "$QS/api/history/get" -H "Authorization: ApiKey $KEY"

6. API reference

Full interactive docs: http://localhost:60610/docs  ·  machine spec: /openapi.json. The endpoints a UI usually needs:

MethodPathPurpose
Status & monitoring
GET/api/statusManager + worker + queue state (poll this)
GET/api/pingLiveness check
GET/api/console_outputRecent RE Manager console text
GET/api/stream_console_outputLive console stream (text/event-stream)
Environment (RE worker)
POST/api/environment/openImport the IOS profile (do this first)
POST/api/environment/closeShut the worker down
Plans & devices (build UI menus from these)
GET/api/plans/allowedAvailable plans + parameter signatures (incl. E_ramp)
GET/api/devices/allowedAvailable devices (incl. sclr, pgm, vortex)
Queue
GET/api/queue/getCurrent queue + running item
POST/api/queue/item/addAdd a plan
POST/api/queue/item/executeRun a plan immediately (bypass the queue)
POST/api/queue/clearEmpty the queue
POST/api/queue/startRun the queue
POST/api/queue/stopStop after current item
Run control
POST/api/re/pausePause the running plan
POST/api/re/resumeResume
POST/api/re/stopStop cleanly
POST/api/re/abortAbort
History
GET/api/history/getCompleted items + results
POST/api/history/clearClear history

Typical UI flow: environment/open → poll status until worker_environment_exists → load plans/allowed + devices/allowed to build forms → queue/item/add (or execute) → queue/start → poll status / stream console_output → read history/get.
