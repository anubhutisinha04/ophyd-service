Frontend ↔ Backend integration guide

Audience: the developer building the periodic-table UI and any other beamline-configuration widgets that need to write values to instrument hardware.

Scope: what the backend provides, what the frontend is responsible for, and exactly how to drive the new batch-caput endpoint for "configure the beamline for edge X" workflows.

1. The mental model

Three layers. Understand these and the rest of the doc is detail.

1.1 EPICS PVs — the bottom layer

A PV (Process Variable) is a single named value living inside an IOC (a controller program running on a beamline computer). Examples:

  • XF:23ID2-ES{CurrAmp:3}Gain:Val-SP — the sample current amplifier's gain setpoint
  • XF:23ID2-ES{Vortex}mca1.PRTM — the vortex MCA's preset counting time

Each PV has a current value, and any program on the EPICS network can read it (caget) or write to it (caput). Configuring an experiment ultimately means doing a batch of caputs to put the right values into the right PVs. Everything else in this doc is plumbing around that single fact.

1.2 Ophyd — the object wrapper

PV name strings are ugly and easy to typo. Ophyd is a Python library that groups related PVs into Python objects. Instead of writing 50 PV strings, you get a tree of attributes:

vortex.mca.preset_real_time.put(0.5)         # corresponds to XF:23ID2-ES{Vortex}mca1.PRTM
vortex.mca.rois.roi2.lo_chan.put(950)        # corresponds to XF:23ID2-ES{Vortex}mca1.R2LO

The shape of the tree is defined by a device class. As a frontend dev you mostly don't deal with ophyd directly — you'll address things by the device's friendly name (and, eventually, by dotted sub-component paths) and the backend resolves to PVs.

1.3 Configuration service — the registry

The configuration service is the device registry. It knows:

  • Which devices and standalone PVs exist on this beamline (its happi_db.json equivalent, plus its CRUD API).
  • Whether each device is currently locked (a plan is running) or disabled (administratively offline).

The configuration service is read by direct-control on every write to validate that the target exists and isn't locked. The frontend usually doesn't talk to the configuration service directly — direct-control does it on your behalf when you make a caput request.

1.4 Direct-control — the write/read service

Direct-control is the service the frontend POSTs to. It:

  • Takes a PV name + value, asks the configuration service "is this PV registered and unlocked?", and if so, issues the caput via pyepics.
  • Same flow for reads (caget), plus WebSocket subscriptions for streaming monitor updates.

Direct-control's API today is PV-name based. You send PV strings, not ophyd dotted paths. Resolving "friendly" addresses (like vortex.mca.rois.roi2.lo_chan) to PV strings is a separate concern — see §7 Follow-ups.

2. The periodic-table workflow

This is the use case the new batch-caput endpoint exists to serve. The scientist clicks an element in a periodic-table widget, and the beamline reconfigures for that absorption edge.

2.1 Sequence

┌────────────┐                  ┌──────────────────────┐
│  Frontend  │                  │   direct-control     │
│ (periodic  │                  │   (port 8003)        │
│   table)   │                  └──────────────────────┘
└────────────┘                            │
       │ 1. Operator clicks "Ni_L"        │
       │ ──────────────────────────────►  │ (no traffic yet)
       │                                  │
       │ 2. Frontend looks up Ni_L in its │
       │    edge-preset table (bundled    │
       │    JSON or fetched from a future │
       │    presets endpoint)             │
       │                                  │
       │ 3. POST /api/v1/pv/set/batch     │
       │ ──────────────────────────────►  │ validates each PV via
       │    { caputs: [12 entries] }      │ config-service, runs
       │                                  │ caputs in order, halts
       │                                  │ on first failure
       │ ◄──────────────────────────────  │
       │   200 OK                         │
       │   { ok: true, applied: 12, ...}  │
       │                                  │
       │ 4. Frontend updates UI:          │
       │    "Configured for Ni_L"         │
       │    (or shows the failure row     │
       │    from the response on error)   │
       │                                  │

2.2 Who owns the edge-preset table

The mapping from edge_name → {PV: value, ...} is the frontend's responsibility. The backend (configuration_service) is a device registry and device-lock store — not a metadata or presets store. Two practical options for where the presets actually live:

OptionHowWhen to pick it
Static JSON bundled with the frontend Ship edge_presets.json alongside the periodic-table widget; rebuild and redeploy to change values. Values rarely change. Simplest possible thing.
Redis (or similar small KV store) Frontend reads/writes xas:presets:<edge> keys against a small presets service. Scientists edit presets routinely without a frontend rebuild. The current xlsx workflow says this is the realistic case.

Either way, the JSON the frontend constructs per click is the same shape and gets POSTed to the same endpoint.

3. The batch caput endpoint

Endpoint: POST /api/v1/pv/set/batch

3.1 Request

{
  "caputs": [
    {
      "pv_name": "XF:23ID2-ES{CurrAmp:3}Gain:Val-SP",
      "value": "1",
      "wait": true,
      "timeout": 2.0
    },
    {
      "pv_name": "XF:23ID2-ES{Vortex}mca1.PRTM",
      "value": 0.5,
      "wait": true,
      "timeout": 2.0
    }
    // ... up to 100 items per batch
  ]
}

Each item is the same shape as a single POST /api/v1/pv/set:

FieldTypeRequiredNotes
pv_namestringyesThe EPICS PV string. Must be registered in configuration_service.
valueanyyesNumber, string (for enums), or array. Type must match the PV.
waitboolno, default falseIf true, block until the IOC reports put-completion. For experiment setup you almost always want true so you know it landed.
timeoutfloatnoSeconds. Used with wait=true or use_complete=true.
use_completeboolnoPower-user knob; use put-callback completion instead of blocking a CA thread. Leave default for most cases.
connection_timeoutfloatnoMax seconds to wait for the EPICS CA connection to come up.

Constraints:

  • At least 1 item, at most 100 items per request (raise this if you have a real need).
  • Extra fields at the top level are rejected (typo-catching).

3.2 Response

{
  "ok": true,
  "applied": 12,
  "requested": 12,
  "results": [
    {
      "pv_name": "XF:23ID2-ES{CurrAmp:3}Gain:Val-SP",
      "success": true,
      "value_set": "1",
      "timestamp": "2026-05-21T14:32:01.123456",
      "coordination_checked": true,
      "mode": "put-completion",
      "status_code": 200,
      "message": null,
      "error_type": null
    }
    // ... one entry per caput attempted
  ]
}
FieldMeaning
oktrue iff every item succeeded. false if any item failed (and the batch halted at that point).
appliedCount of items that succeeded before the halt (or all of them on success).
requestedTotal items in the batch you sent.
resultsOne entry per item attempted. Items past the failure point are absent — that's how you know where the batch halted.

Per-item status_code mirrors what the equivalent single POST /api/v1/pv/set call would have returned: 200 on success, 404 for unknown PV, 423 for locked device, 409 for disabled device, 503 for coordination-service trouble, 500 for an unexpected error.

3.3 Fail-hard semantics

The batch halts on the first failure. Items past the failure point are NOT attempted. Items already applied are NOT rolled back — the IOC has no concept of a transaction, so the partial state stays in place. This is intentional: continuing past a failure risks leaving the instrument in a worse, more inconsistent state.

How to handle a partial-batch failure on the frontend:

  1. Check ok. If true, you're done.
  2. If false, the last entry in results is the failure. Look at its error_type, message, and status_code to decide what to show.
  3. The operator should see the failure — don't silently move on. The instrument is now in a half-configured state.

3.4 HTTP status of the batch call itself

The batch call returns 200 on any well-formed request — even when items inside fail. This is deliberate so you can always read the structured results list. Non-2xx from this endpoint means the request itself was malformed (422 for empty list, bad value type, extra fields, etc.) or the service itself is down.

4. The single caput endpoint (still available)

For one-off sets (e.g. a slider widget that nudges one motor), use the existing single endpoint:

POST /api/v1/pv/set
{ "pv_name": "...", "value": ..., "wait": true, "timeout": 2.0 }

It returns the per-item shape directly (not wrapped in a batch envelope) and uses standard HTTP status codes (200 / 404 / 423 / 503 / 500).

5. Reading values (caget)

One-shot read of a PV:

GET /api/v1/pv/{pv_name}/value?use_monitor=false&timeout=2.0

Response is a tiled-style envelope with value, shape, dtype, units, alarm fields, etc. For streaming updates use the WebSocket endpoints (/api/v1/pv-socket and /api/v1/device-socket) — see the direct-control service README for those.

6. End-to-end example: configuring for Ni_L

Suppose the frontend's edge-preset table for Ni_L looks like:

{
  "Ni_L": {
    "sample_gain":     "1",
    "sample_decade":   "1 nA/V",
    "aumesh_gain":     "1",
    "aumesh_decade":   "1 nA/V",
    "pd_gain":         "5",
    "pd_decade":       "1 uA/V",
    "sclr_time":       0.5,
    "vortex_preset":   0.5,
    "au_mesh_y":       -130.2,
    "m1b1_pid_sp":     378,
    "epu1_offset":     0.2,
    "epu_table":       4
  }
}

The frontend resolves each friendly key to the PV string (from configuration_service or a local mapping) and POSTs:

POST /api/v1/pv/set/batch
{
  "caputs": [
    {"pv_name": "XF:23ID2-ES{CurrAmp:3}Gain:Val-SP",     "value": "1",       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{CurrAmp:3}Gain:Decade-SP",  "value": "1 nA/V",  "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{CurrAmp:2}Gain:Val-SP",     "value": "1",       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{CurrAmp:2}Gain:Decade-SP",  "value": "1 nA/V",  "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{CurrAmp:1}Gain:Val-SP",     "value": "5",       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{CurrAmp:1}Gain:Decade-SP",  "value": "1 uA/V",  "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{Sclr:1}.TP",                "value": 0.5,       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-ES{Vortex}mca1.PRTM",          "value": 0.5,       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID2-BI{AuMesh:1-Ax:Y}Mtr.VAL",     "value": -130.2,    "wait": true, "timeout": 30.0},
    {"pv_name": "XF:23ID2-OP{FBck}PID-SP",               "value": 378,       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID-ID{EPU:1-FLT}Val:InpOff1-SP",   "value": 0.2,       "wait": true, "timeout": 2.0},
    {"pv_name": "XF:23ID-ID{EPU:1}Val:Table-Sel",        "value": 4,         "wait": true, "timeout": 2.0}

On success the UI flashes a "Configured for Ni_L" indicator. On failure it shows which caput failed (PV name + error message) and tells the operator the instrument is in a partial state.

7. Follow-ups (not yet wired up)

Dotted-path addressing. Today the frontend has to know the raw PV string for every preset value. We've discussed adding a "resolve" endpoint to configuration_service that takes a friendly address like vortex.mca.rois.roi2.lo_chan and returns the underlying PV string by walking the ophyd device class's Component tree. Once that lands, the frontend's preset table can use friendly addresses and the resolve call becomes part of the per-click flow (or once-at-load-time, with caching). Not in this PR.
Presets storage. Where the per-edge preset table actually lives (frontend bundle vs redis vs something else) is an open call — see §2.2. This is a frontend decision; the backend doesn't constrain it.
Plan kwargs (out of scope for caput). Some columns in the existing xlsx (scan_count, intervals, and arguably start / stop / velocity for fly scans) are not PVs at all — they're bluesky plan kwargs that the queueserver consumes when running a scan. Those don't go through this endpoint. When/if the queueserver integration lands, the frontend will need a separate "submit plan" API for them.

8. Things to watch out for

  • Don't silently swallow ok: false. If the batch halted partway, the operator needs to know — the IOC is in a partial state.
  • Use wait: true for setup caputs. Fire-and-forget (wait: false) is fine for "nudge this slider" UIs but bad for "configure for an edge" — you want to know the values actually landed before the operator hits "start scan".
  • Locks are first-class. When a plan is running, every device it owns will return 423 Locked. Surface this clearly in the UI so the operator knows to wait or stop the plan, not retry.
  • Empty batches are a frontend bug. The endpoint rejects them with 422 rather than silently doing nothing.
