queueserver_service

The plan runner — takes bluesky plans, queues them, executes them in a worker process, streams progress back to the UI.

How this service came to be

queueserver_service is one of the two backends that started life as someone else's code. The upstream world has two separate packages — bluesky-queueserver (the RE manager + worker) and bluesky-httpserver (a FastAPI front-end for it) — that together do what this one service now does. The ophyd-service fork vendored both packages in, merged them into a single cohesive Python package, and has been refining and simplifying that merger ever since.

Phase A · Vendor + strip (2026-06-10)

Upstream is two separate PyPI packages that have to be installed together to be useful. Rather than run against those PyPI packages as black boxes, the fork imported both — as a squash of the upstream tree — so the code could be modified in-tree:

Commit 356d521 — "feat: add queueserver_service backend, based on bluesky-queueserver + bluesky-httpserver"
Commit 22da9b3 — "chore(queueserver): strip upstream-repo artifacts (CI workflows, repo templates, deploy key, codecov)"

The compatibility gate for the bluesky-queueserver-api PyPI client was set up in the same window (a1ae8cd) so every subsequent change could be measured against "does the real client still work?".

Phase B · Unify (2026-06-10)

The imported code inherited upstream's two-package layout — bluesky_queueserver/ plus subprojects/bluesky-httpserver/, two setup.py files, two sets of test suites, two docs trees. Commit 149578e replaced that with a single package, keeping every external contract intact. From its own message:

Commit 149578e — "queueserver_service: restructure into one cohesive package.
Replace the two stapled-together upstream repo layouts (bluesky_queueserver + subprojects/bluesky-httpserver) with a single queueserver_service package … console-script names, wire protocols, cookie names and the locked_by_service default are unchanged; OpenAPI schema regenerated and diffs only in the version field."

Result:

  • queueserver_service/manager/ (was bluesky_queueserver.manager)
  • queueserver_service/http/ (was bluesky_httpserver)
  • queueserver_service/common/ — the comms/JSON-RPC/logging code that both halves shared
  • One pyproject.toml replaced two setup.py/setup.cfg/versioneer/MANIFEST duplicates
  • tests/manager/ + tests/http/ collected by one pytest run

Phase D · HTTP-only (in progress — this orientation shows the target)

The largest simplification is still landing. Upstream bluesky-queueserver speaks two protocols to the outside world: HTTP (for browsers) and 0MQ (for CLI tools and older clients). The ophyd-service target state removes 0MQ entirely, leaving HTTP + WebSocket as the single external contract. Manager↔worker communication (multiprocessing pipes) is unaffected — that's internal and not "on the wire".

Where Phase D stands today

  • Done: the structural prerequisites — dispatch serialization lock so HTTP handlers can't race the 0MQ REP loop (PR #83); the Side-C compat test suite that pins the HTTP contract (PR #91); WS auth parity with the HTTP path (PR #95); the bluesky-httpserver shim (PR #88).
  • Planned in the current remediation: in-process asyncio bridge to replace the 0MQ PUB console/info stream (stage 6b), rework the qserver operator CLI onto the HTTP transport of bluesky-queueserver-api (stage 6c), then delete the REP loop, PUB publisher, zmq halves of common/comms.py, CLI flags, and QSERVER_ZMQ_* env vars (stage 6d).
  • Guardrail: the Side-C compat suite must pass identically before and after every stage. If a real REManagerAPI HTTP-transport call regresses, the stage doesn't ship.

The rest of this page describes the target state after Phase D lands — one process, one HTTP + WS contract, no 0MQ on the wire. When you read the current code and find residual 0MQ imports (manager.py, output_streaming.py, common/comms.py), you're looking at code slated for deletion in stage 6d.

The picture

What it does (in plain English)

Holds a plan queue

Users submit bluesky plans (scan, count, custom). The manager stores them in a persistent queue. Three queue-store backends are plug-selectable via URI: redis:// (the historical default), sqlite+aiosqlite:///, and postgresql+asyncpg://. Plans wait until the queue is started and the environment is open.

Runs plans in a worker

A separate RE Worker subprocess actually runs the bluesky RunEngine. The manager talks to it over pipes. The worker imports a "profile collection" that defines devices and plans.

Streams progress to the UI

Console output, system info, and status updates are pushed over WebSocket. The React UI sees plan-by-plan progress and any print statements from the plan.

Coordinates with configuration_service

Before starting a plan, the manager locks the devices it needs (see Side B). It also keeps its device list in sync with the registry via a periodic diff.

Unified mode: one process, one API

The HTTP-only pivot

Why this is a big deal

Upstream bluesky-queueserver exposes two protocols: HTTP (for the browser) and 0MQ (for the qserver CLI and older clients). The ophyd-service fork removes 0MQ entirely and keeps HTTP as the single external contract. Fewer moving parts, one thing to test, one thing to secure.

Target-state improvements

Manager can't silently brick anymore

The worker-state poll task used to die permanently on a double-set_result — the manager kept running but its view of the worker was frozen forever. Every future set is guarded now, and the loop body itself is wrapped so a single exception can't kill the polling task.

delivered

No more duplicate plan starts

In unified mode, HTTP and (legacy) 0MQ handlers could both pass the "queue is idle" check and both start a plan. A single asyncio.Lock around dispatch guarantees one command at a time. This is also the structural prerequisite for going HTTP-only.

delivered

Plan reports and task results survive

Plan reports used to be destroyed on read with a 0.5 s pipe timeout — a slow read would mark a completed plan as failed and re-execute it. Reports are now retained until explicitly reset, and the fetch uses the long timeout with one retry. Task results are retained until the caller acknowledges receipt.

delivered

SIGKILL-ed worker is detected

Previously a killed worker left the manager stuck in EXECUTING_QUEUE forever. After N consecutive state poll timeouts the manager now checks the watchdog; on a positive "worker dead" it recovers cleanly. A watchdog comm timeout is treated as "unknown", so a stalled host can't false-positive.

delivered

Overlay updates are atomic

Applying a registry overlay in the worker used to run partway and leave a half-updated namespace on failure. It's now stage-then-commit: every upsert instantiates into a staging dict; only if all succeed does the worker's namespace mutate. Uses the long pipe timeout, so a big registry change doesn't abort the running plan.

delivered

Guarded config-service polling

The periodic device-sync poll now runs inside a guard that logs failures and surfaces them in the status response, instead of dying silently as an orphaned task. If configuration is unreachable at manager restart, the manager degrades rather than crash-loops.

delivered

HTTP-only, in-process console

The output console will be delivered by an in-process asyncio queue from the worker into the HTTP layer, replacing the 0MQ PUB socket. Bounded queue with drop-oldest so a slow client can't back up the worker.

in progress

qserver CLI on HTTP

The qserver operator CLI is being reworked as a thin wrapper over the HTTP transport of bluesky-queueserver-api (which is already a dependency). Same commands, no 0MQ required.

in progress

Modular manager + routers

The 4700-line manager.py has already had its config-service coordinator client extracted (~914 lines in manager/config_service.py), and the HTTP routes have been split into 12 domain routers under http/routers/ (admin, console, environment, execution, locks, permissions, plans_devices, profile_collection, queue, run_engine, status). Further splits (lock_info, worker_comm, handler modules) happen after 0MQ removal so we don't split code about to be deleted.

delivered / remainder planned

Compatibility contract — a hard requirement

The frozen contract queueserver must keep is the HTTP + WebSocket surface consumed by bluesky-queueserver-api's HTTP transport. It is enforced by the Side-C compat test suite, which drives the real PyPI client library against a live server on every PR.

Compatibility with bluesky-queueserver-api

The single most important non-negotiable in the whole queueserver evolution is this: the PyPI bluesky-queueserver-api client must keep working against our service. Every simplification, every refactor, every 0MQ-removal stage has to pass that test. Without it, downstream consumers built on the API library break.

The catch is that the API library ships two transports under the same top-level package:

An importer of the library writes exactly one of:

from bluesky_queueserver_api.http import REManagerAPI   # HTTP transport
from bluesky_queueserver_api.zmq  import REManagerAPI   # 0MQ transport

Both classes expose the same methods (status(), queue_start(), environment_open(), item_add(), …). Only the wire protocol differs. The ophyd-service target keeps the HTTP one working. The ZMQ one continues to work against upstream deployments, just not against ours.

Why this matters — a real downstream: BLOP

BLOP (the Bluesky Optimization Package) is a concrete example of an unrelated project that depends on bluesky-queueserver-api to drive experiments:

  • BLOP declares queueserver = ["bluesky-queueserver-api"] as an optional dependency in its pyproject.toml.
  • Its QueueserverOptimizationRunner accepts a re_manager_api argument — an instance of REManagerAPI from either transport. BLOP's own docs are explicit: "The re_manager_api argument also accepts an HTTP-based client (bluesky_queueserver_api.http.REManagerAPI) for deployments that expose the queueserver over HTTP rather than ZMQ."
  • BLOP's example code today constructs the .zmq variant. Against ophyd-service's HTTP-only queueserver_service, a BLOP user just imports from .http instead — the rest of their integration doesn't change.

What downstream consumers actually need to do

If you're a downstream that currently uses bluesky_queueserver_api.zmq.REManagerAPI against an ophyd-service deployment: swap the import to bluesky_queueserver_api.http.REManagerAPI and construct it with the HTTP URL + API key instead of the ZMQ address. That's the whole migration. If you're already on the HTTP transport, nothing changes for you at all.

How the compat guarantee is enforced

Three mechanisms keep the HTTP-transport contract from silently regressing:

  1. Distribution + shim naming. This service's dist is named bluesky-queueserver so the API client's Requires-Dist resolves in-tree (not to upstream, which would shadow our console scripts). A companion bluesky-httpserver shim distribution provides the import name and dist name upstream code expects.
  2. Legacy import surface. A thin top-level bluesky_queueserver package re-exports the names the API client (and existing beamline startup scripts) reach for — bluesky_queueserver.manager.annotation_decorator, .profile_tools, and friends — mapped onto the in-tree queueserver_service implementation.
  3. Side-C test suite (the acceptance gate). Drives the real PyPI bluesky_queueserver_api.http.REManagerAPI against a live queueserver_service on every PR. Every 0MQ-removal stage has to pass it identically before and after. If a call the client makes stops working, the stage doesn't ship.
