Frontend integration

How the React + Finch UI talks to the three backend services. Owned by another team — this page describes what the backend exposes to them, not their code.

The picture

What the frontend gets from each service

From configuration_service

Metadata + admin. The list of devices and PVs, who owns what, whether devices are enabled. Also the admin operations for adding/editing/enabling/disabling devices from the UI.

Access pattern: REST only. No WebSocket — the registry changes rarely enough that polling or refresh-on-action is fine.

From direct_control_service

Live values and hardware control. Widgets subscribe to PVs or devices over WebSocket for real-time values; they POST individual writes or a batch of writes ("configure beamline for edge X" = 40 PVs in one round-trip).

Access pattern: WebSocket for streaming, REST for one-shot reads and writes.

From queueserver_service

Plan lifecycle. Submit a plan, see the queue, watch the console output stream, view history. Also environment open/close and RE pause/resume/stop controls.

Access pattern: REST for queue edits & commands, WebSocket for status & console streams.

OpenAPI-driven TypeScript

Every backend publishes its full OpenAPI spec to shared-schema/<service>.openapi.json. The frontend runs openapi-typescript to turn each schema into .d.ts files that give type checking on every route, parameter, and response body.

The commands, from the top-level README:

npx openapi-typescript shared-schema/configuration_service.openapi.json \
    -o frontend/src/api/configuration_service.d.ts
npx openapi-typescript shared-schema/direct_control.openapi.json \
    -o frontend/src/api/direct_control.d.ts
npx openapi-typescript shared-schema/queueserver_service.openapi.json \
    -o frontend/src/api/queueserver_service.d.ts

WebSockets are not in the OpenAPI spec

FastAPI doesn't emit WebSocket endpoints into the OpenAPI schema. Each service documents its WebSocket contracts prose-style in the service README — see direct_control_service/README.md and queueserver_service/README.md. The frontend team writes the WS message types by hand.

Development topology

The WebSocket contracts, in brief

direct_control — three WebSocket endpoints

EndpointClient saysServer pushes
/ws/pv-socket { action: "subscribe", pvs: [...] }
{ action: "subscribeReadOnly", pvs: [...] }		 subscribed per PV that stuck
pv_subscribe_failed per PV that didn't
pv_update with value + alarm + timestamp
/ws/device-socket { action: "subscribe", devices: [...] } updates keyed by device name, one push per component PV
/ws/control-socket { action: "set", pv: ..., value: ... } per-write acks (also coordination-gated)

queueserver — three WebSocket endpoints

EndpointPurpose
/api/status/wsManager state updates (queue length, running plan, env state, sync errors).
/api/console_output/wsEverything the plan or worker prints, line by line.
/api/info/wsSystem info (memory, uptime, versions) at a lower cadence.

Auth story

configuration_service

No in-service auth today for read routes. Destructive admin routes (force-unlock, reset) will be gated by a shared secret — that work is planned in the current remediation.

direct_control_service

No in-service auth. Authorization is expected from upstream middleware (reverse proxy, API gateway). A global read-only mode (DIRECT_CONTROL_GLOBAL_READ_ONLY=true) refuses every write, useful for monitor-only deployments.

queueserver_service

Has the full upstream bluesky-httpserver auth stack: API keys, session tokens, external OIDC providers, DictionaryAuthenticator for dev. WebSocket auth accepts Authorization: ApiKey ... (case-insensitive) and ?api_key=...; cookies deliberately rejected on WS to avoid CSWSH.

What the frontend team can rely on (target state)

  • One HTTP surface per service, one WebSocket surface per service. No 0MQ anywhere on the target-state backend.
  • Types generated from OpenAPI stay in sync with the running server — a CI drift test enforces this.
  • Widget writes fail explicitly. 423 Locked, 422 out of range, 409 UNKNOWN gate. No silent no-ops.
  • Live values carry alarm severity. UIs can render MINOR / MAJOR / INVALID without a second query.
  • Errors on WS subscribes are surfaced. If a PV doesn't stick you get pv_subscribe_failed, not silence.
  • Devices are never permanently locked because queueserver crashed — TTL leases release automatically.
