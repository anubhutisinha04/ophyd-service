Side A · configuration ↔ direct_control

The conversation between the source-of-truth registry and the service that actually talks to hardware. Everything a user does with a UI widget travels this seam.

The picture

The classic user write — full sequence

The single most important flow on Side A is what happens when a user clicks "Set" in the UI. Here it is, gate by gate:

The four conversations, in detail

1 · Registry lookup — "what is this thing?"

Whenever direct_control needs to write a PV or run a device method, it first asks configuration for the metadata:

  • GET /api/v1/pvs/{pv} — who owns this PV? what's its type?
  • GET /api/v1/devices/{name} — device details, PV list, whether it's enabled.
  • GET /api/v1/devices/{name}/instantiation — Python class path and constructor arguments to build the ophyd object (device-method path only).

Direct_control caches nothing durable — the registry is the truth, every lookup is a live call. Configuration returns 503 while it's still starting; direct_control's startup deliberately blocks on configuration's /health before serving traffic.

2 · Coordination gate — "am I allowed?"

The safety-critical call. Before any write reaches EPICS, direct_control hits GET /api/v1/devices/{name}/status and inspects lock_status:

lock_statusWhat direct_control does
unlockedProceed to the EPICS put.
locked (by me)Proceed. (The write is coming from the entity that holds the lock.)
locked (by someone else)Return 423 Locked. Do not touch EPICS.
UNKNOWN (any unmodelled response)Return 409 Conflict. Fail-closed.

The gate is opt-outable via DIRECT_CONTROL_COORDINATION_CHECK_ENABLED=false for testing, but in production it's always on.

3 · Enrichment — the reverse direction

The one call where configuration acts as the client. When configuration is (re)building its registry from a profile, it needs to know each device's live metadata — connect status, PV list, alarm state. It POSTs a batch of device specs to POST /api/v1/devices/enrich on direct_control, which connects each one and returns the enriched view.

Configuration handles a direct_control outage gracefully: the batch is marked enrichment_unavailable instead of failing the whole registry load.

4 · PV health reports

Every EPICS operation direct_control performs — read, write, monitor callback failure — is reported back to configuration:

  • POST /api/v1/pvs/{pv}/success
  • POST /api/v1/pvs/{pv}/failure (with an error description)

These are fire-and-forget: the outcome of the user's write is never blocked on the health report succeeding. Configuration aggregates the reports into per-PV counters + last-seen timestamps so a UI can highlight failing hardware.

What can go wrong (and what changed)

Configuration is down

Behaviour: direct_control returns 503 to the user (registry lookup fails). The safety gate fails closed — no writes go through. Startup blocks on configuration's /health, so a fresh direct_control won't come up if configuration isn't ready.

Configuration returns UNKNOWN status

New behaviour: the write returns 409 Conflict (a coordination problem) and no PV-health report is sent. A gate failure was never about the PV, so it must not affect the PV's health score.

Direct_control connects a dead PV

Old behaviour: the 5-second EPICS connect held the global monitor lock, freezing every other subscriber.
New behaviour: per-PV connect lock, connect happens outside the fan-out lock, dead PVs never freeze live ones.

A client disconnects mid-subscribe

Old behaviour: the EPICS monitor stuck around forever.
New behaviour: after the subscribe returns we re-check under the lock that clients still want the PV; if not we unsubscribe immediately. Both sockets got the fix.

Configuration flaps briefly

The queueserver-side httpx client has retries baked in; the direct_control clients are being brought onto the same retry helper as part of the shared-package work, so a transient blip mid-session stops turning every write into a 503.

Registry write races

Old behaviour: concurrent create-same-name requests could both pass the "does it exist?" check and both write.
New behaviour: one app-wide asyncio lock serializes every registry mutation; concurrent same-name creates yield exactly one 201 and one 409.

The mental model

Two rules

  1. Direct_control never writes without first asking configuration. That's the safety story — the gate can't be bypassed by any code path.
  2. Configuration never touches hardware directly. When it needs live device metadata (enrichment), it asks direct_control. When it wants to know if a PV is healthy, it consumes reports — it doesn't ping.
