configuration_service

The source of truth for what devices exist on the beamline and who is allowed to touch them right now.

The picture

What it does (in plain English)

Keeps a device list

Every device on the beamline is registered here with its Python class path, constructor arguments, and the PVs it owns. Four loader strategies (CONFIG_LOAD_STRATEGY): happi (parses happi_db.json), bits (parses BITS devices.yml), mock (built-in sample data for tests), empty (start with nothing and register via REST). Default is auto, which picks the first strategy whose inputs are present.

Enforces one writer per device

When queueserver is about to run a plan, it asks configuration to lock the devices it needs. Any write attempt from another source (direct_control, another operator) is refused until the lock is released.

Remembers PV health

Direct_control reports the outcome of every PV read/write. Config aggregates those into a per-PV health record — how many successes, how many failures, when it last worked — so the UI can highlight dead hardware.

Publishes an OpenAPI schema

Every route is documented in a machine-readable schema written to shared-schema/configuration_service.openapi.json. The frontend turns this into TypeScript types.

Key API endpoints (in plain English)

RouteWhat it's forWho calls it
GET /api/v1/devices List every registered device. UI, queueserver, direct_control
GET /api/v1/devices/{name} Look up one device: PVs, metadata, whether it's enabled. direct_control (on every write)
GET /api/v1/devices/{name}/instantiation Give me the Python recipe to build this device (class + kwargs). direct_control (device-method path), queueserver
GET /api/v1/devices/{name}/status Is it locked? By whom? Since when? direct_control (safety gate on every write)
POST /api/v1/devices/lock Take a lock (with an owner + TTL lease). queueserver, before starting a plan
POST /api/v1/devices/lock/renew Heartbeat — extends the lease on locks held by this item_id. Returns which devices were renewed vs lost vs conflict. queueserver (only meaningful when leases are enabled)
POST /api/v1/devices/unlock / force-unlock Release a lock cleanly, or forcibly (admin). queueserver on plan end · operator on cleanup
POST /api/v1/devices/enrich "Fill in these device specs with live metadata." Reverse call — config asks direct_control. configuration → direct_control
POST /api/v1/pvs/{pv}/success / /failure Fire-and-forget health report. direct_control after every EPICS operation
GET /health Liveness probe. Returns 503 until warmup is complete. docker, direct_control at startup, humans

What runs where

Target-state improvements

The 2026-07-02 backend review flagged several safety and integrity gaps in configuration_service. The following are the target-state improvements this orientation reflects:

All registry mutations serialized

Every create / update / delete / reset now holds a single app-wide asyncio lock, closing check-then-act races between concurrent requests. Running with --workers > 1 is rejected at startup because the in-memory state can't span workers.

delivered

Lock lifecycle can't leak

Deleting or updating a locked device is now refused (409). If a device is removed from the registry with its lock still held, a force-unlock cleanly releases the orphan. A registry reset or clear wipes the lock table — no more "beamline shows all-locked after a reset".

delivered

Locks with optional TTL leases

Every lock can carry a TTL. If it does, the lock holder must send heartbeats via POST /devices/lock/renew; if heartbeats stop (crash, network partition, kill), the lease expires and the device becomes writable again automatically. Leases are opt-in (CONFIG_LOCK_LEASE_TTL_SECONDS, default 0 = disabled). With leases disabled, locks are held until an explicit release or a force-unlock — the historical behaviour.

delivered

Lock-authority epoch (always on)

Independent of the lease feature, every configuration_service process start generates a fresh lock_epoch, surfaced on lock/renew/status responses. Distinct from the DB-backed registry service_epoch: a plain restart drops all in-memory locks but preserves the registry, so the lock_epoch is what signals lock-table loss to queueserver and direct_control.

delivered

Force-unlock is atomic

Force-unlock now validates the full request before touching any lock. Either every named device is released or none are; the audit row is always written. No half-applied force-unlocks.

delivered

Input validation hardened

Device names, PV names, and standalone-PV values are all validated for printable-ASCII / non-empty / no path separators. An opt-in CONFIG_DEVICE_CLASS_ALLOWLIST refuses to import unknown Python class paths.

delivered

Event loop stays responsive

Blocking work — loading a big profile file, importing device classes — now runs off the event loop via asyncio.to_thread, so a slow disk or a big BITS profile no longer stalls unrelated requests.

delivered

/health never leaks DB internals

The 503 response body no longer echoes raw DB exceptions (which could leak DSN / host / credentials). It reports "not ready" with a safe category only.

delivered

Shared-secret auth on admin routes

Destructive endpoints (force-unlock, registry reset) will be gated by a shared secret so they can't be hit from an unauth'd process on the same host.

planned

Rule of thumb

configuration_service never touches EPICS. It doesn't read a PV, it doesn't move a motor. It just answers the questions "what do we have?" and "who owns it?", and remembers what direct_control tells it about PV health.
