Key updates that shaped the target state

A curated tour of the most consequential changes on the way to the target-state backend. Not every PR — just the ones that meaningfully changed what a new reader needs to understand.

How to read this page

Grouped by theme, not phase or date. Each entry names the shipped behaviour first and (in a subline) the concrete PR that landed it, so you can find the diff. Legend: delivered lands are on main; in flight is an open PR; planned is scoped but not yet in flight.

Theme 0 · The queueserver_service journey

Before any of the reliability/safety fixes below, there was a different kind of change: turning the two upstream packages into one refined service. These commits set up everything else on this page and are worth reading first.

What happenedWhenLanded in

Why this matters

Nearly all of the other changes on this page happen in code that was upstream's, is now the fork's, and is on a trajectory toward "HTTP-only, one cohesive package". Reading a change without that context makes it look like a random pile of fixes; reading it with that context makes it a coherent story of gradually simplifying a two-package vendor drop into one clean service.

Theme 1 · Reliability wins

What changedBeforeAfterLanded in
Lock leases + lock-authority epoch delivered Locks were indefinite in-memory with no expiry; a queueserver crash left every held device locked until an operator force-unlocked (or configuration_service restarted, which itself dropped every lock silently — direct_control couldn't distinguish "unlocked" from "authority reset"). Opt-in TTL leases (CONFIG_LOCK_LEASE_TTL_SECONDS, default 0 = legacy) with a POST /devices/lock/renew heartbeat: crashed holders lose their locks after the TTL. An always-on lock_epoch (fresh per configuration_service process start, distinct from the DB-backed registry service_epoch) surfaces on lock/renew/status so queueserver re-acquires and direct_control logs an attributable "authority reset" event. Queueserver's lock_scope default also changed from environment to plan. PR #81
Manager silent-brick fixed delivered The worker-state poll task died permanently on a double set_result; the manager kept running but its view of the worker was frozen forever. Watchdog couldn't see it (heartbeats live in another task). Every future set is guarded with if not fut.done(): via a new _complete_manager_task. The loop body is wrapped so exceptions don't kill the task. PR #83
Dispatch serialization lock delivered HTTP and 0MQ handlers could concurrently start plans; two concurrent queue_start calls could both pass the IDLE check. One asyncio.Lock in _dispatch_command serializes every command path. Also the structural prerequisite for going HTTP-only. PR #83
Worker-death detection delivered A SIGKILL-ed worker left the manager stuck in EXECUTING_QUEUE forever. N consecutive state-poll timeouts trigger a watchdog check. Positive "dead" → recover cleanly. Watchdog comm-timeout treated as "unknown" so a stalled host can't false-positive. PR #84
Plan report retention delivered Plan reports were destroyed on read with a 0.5 s pipe timeout — a slow read marked completed plans as failed and re-executed them. Reports retained until an explicit reset; fetch uses the long pipe timeout with one retry; a delivered-flag gates re-processing. PR #83
Task-results acknowledgement delivered Task results were destroyed on read; a lost pipe response lost the result. Manager sends the task_uids it received last time; worker drops those and returns the rest. Single-flight to avoid flooding the pipe. PR #89
Atomic device overlays + guarded sync delivered Overlay RPC used the 0.5 s pipe timeout while the worker did a full regeneration → every registry change aborted the plan. Partial failures left a half-updated namespace. Periodic sync died silently as an orphan task; manager-restart crash-looped if config was down. Overlay uses the long pipe timeout; worker applies upserts stage-then-commit inside a staging dict. Periodic sync and manager-restart paths both wrapped in a guard that surfaces failures in /api/status. PR #96
Config-registry lock lifecycle delivered A locked device could be deleted (orphan lock, unclearable); registry reset didn't clear locks; force-unlock was non-atomic; enable/disable mutated memory before persisting; PV re-homing was broken on update_device; concurrent same-name creates could double-write. DELETE/PUT on a locked device returns 409; force-unlock is all-or-nothing; reset/clear wipes the lock table; enable/disable copy-then-persist-then-apply; PV ownership re-homing extracted; one app-wide asyncio.Lock around all mutations; --workers > 1 rejected. PR #90, PR #98
Monitor lock + subscribe leak delivered 5-second EPICS connect for a dead PV held the global monitor lock, freezing every subscriber. Client disconnect during in-flight subscribe leaked permanent CA monitors. Per-PV connect lock; connects happen outside the fan-out lock; batch subscribes are concurrent. Both sockets re-check under the lock after subscribe returns and unsubscribe if no clients remain. PR #93

Theme 2 · Hardware-safety wins (direct_control)

What changedBeforeAfterLanded in
Stop-on-command-timeout delivered If motor.set(x) timed out, hardware kept moving with no stop(). Both classic and async drivers issue stop() on timeout. Error message reports whether stop was issued. Opt out per-request or via stop_on_command_timeout=false. PR #97
Coordination status & alarm propagation delivered UNKNOWN coordination status returned 500 and reported PV health (a gate failure treated as a hardware failure). Alarm severity was hardcoded 0/0 in every update. read_only subscription flag was cosmetic — a writable signal was always built. UNKNOWN → 409, no PV-health report. Alarm severity/status plumbed through every update (initial + running). read_only plumbed all the way to the ophyd signal. Failed subscribes send a pv_subscribe_failed envelope. PR #99
Standalone-mode device socket delivered The device socket bypassed the registry abstraction with a raw httpx call and was broken in file/auto mode (every subscribe → upstream_unreachable). Device socket resolves PVs via the same RegistryProvider as the write path (HTTP client or file loader). Standalone/file mode works end-to-end. PR #100 in flight
Control-limit validation delivered No limit validation on any write path (PVLimits/ValueLimitError models existed but were unreferenced). DeviceController reads (lower_ctrl_limit, upper_ctrl_limit) via pv.get_ctrlvars(); out-of-range numeric writes → 422 before EPICS is touched. Fail-open on metadata failure; skipped for non-numerics and EPICS unlimited convention. Opt out per-request or globally. PR #101 in flight

What none of this changes

The invariants worth remembering

  • configuration_service is the source of truth for devices, PVs, and locks. Everything else consults it.
  • direct_control never writes without permission. The coordination gate is on every write path.
  • queueserver runs plans in a separate worker process. The RE lives in the worker; the manager doesn't touch hardware directly.
  • The HTTP + WebSocket surface consumed by bluesky-queueserver-api's HTTP transport is a frozen contract, enforced by the Side-C test suite. 0MQ removal must not regress it.
  • Lock-authority epoch is always on; TTL leases are opt-in. With leases enabled, a crashed holder's locks self-release; with the default (disabled), the lock_epoch signal still lets queueserver re-acquire after a config-service restart.
