Orientation to the backend

A plain-English tour of the three services that make up the ophyd-service backend, how they talk to each other, and how the React/Finch frontend uses them. Reflects the target state after the ongoing backend remediation lands.

How to read this. Every page starts with a picture. Read the picture first; the words underneath just explain what you're looking at. Boxes are always coloured the same way:
configuration_service direct_control_service queueserver_service React + Finch frontend

The 30-second picture

The one-line summary: Configuration is the book (what devices exist and who is using them), Direct-control is the hand (moves one thing right now), Queueserver is the conductor (runs a scripted experiment made of many moves). The React frontend talks to all three.

Why "ophyd-service"?

The project name isn't just decoration. "ophyd-service" is a mental model, not a term the code uses, but the code makes the model real in exactly one place: direct_control_service is the only service in the backend that actually instantiates ophyd device classes and drives them. Everything else works with metadata, PV strings, and lock records.

What that means in practice:

  • The other two services never import ophyd. configuration_service stores class paths and constructor kwargs; queueserver_service passes them to a bluesky RunEngine running in a separate worker.
  • direct_control has both device frameworks as declared dependencies (ophyd + ophyd-async[ca]) and drives each with a dedicated driver. Its src/direct_control/drivers.py is the one place that branches on framework — everything above stays framework-agnostic.
  • Verified from the code: a device class that is neither classic ophyd nor ophyd-async is a hard error. There is no PV-gateway fallback at the device-control level. The service means it when it says "device server".

Two device frameworks, one HTTP surface

direct_control drives both device frameworks, and its own source is explicit about it — from src/direct_control/drivers.py:

"direct_control drives BOTH classic ophyd (pyepics transport) and ophyd-async (aioca/p4p transports) devices during the multi-year migration between the two frameworks."

Two transports, two frameworks, one HTTP request shape. See the direct_control page for the verified layer stack, the two-driver dispatch mechanism, and the git-history evidence of the "PV wrapper → real device server" re-orientation.

Meet the services

configuration_service

The source of truth. Holds the list of every device on the beamline, every PV those devices own, and who currently holds the "lock" on each device. Persists to PostgreSQL (or SQLite in dev).

Nothing moves hardware here — it just knows what exists and who is allowed to touch it.

Read the tour →

direct_control_service

The live wire to hardware — where ophyd lives. Instantiates ophyd (classic and async) device classes on demand, runs their methods, streams live values over WebSockets. Also supports raw PV reads/writes as a lower-level fallback. Every write first checks the lock in configuration.

This is what a UI widget talks to when a user drags a slider or clicks a button.

Read the tour →

queueserver_service

The plan runner. Accepts scripted bluesky plans, queues them, and runs them in a dedicated worker process. Streams console output and status updates back over WebSocket.

Started as a merge of upstream bluesky-queueserver + bluesky-httpserver; unified into one cohesive package; the target state removes 0MQ entirely, leaving HTTP + WebSocket as the single external contract. Compatibility with the bluesky-queueserver-api HTTP transport is a hard requirement so downstream consumers (e.g. BLOP) keep working.

How they talk to each other

There are only two conversations between the backend services, and one implicit one at the hardware layer:

Side A · details →

configuration ↔ direct_control

Every read/write direct_control performs consults configuration first, for both the device metadata (where's the PV? what's its type?) and the safety gate (is anyone else using this right now?). Direct_control also reports PV health back so operators see failing hardware.

Side B · details →

configuration ↔ queueserver

When queueserver spins up a worker, it syncs its device list from the registry, then acquires per-plan locks on the devices the plan needs. An always-on lock_epoch tells queueserver when configuration_service restarts (and drops the in-memory lock table), so it can re-acquire. Opt-in TTL leases add auto-expiry as a third safety net for crashed lock holders.

How the frontend uses all of this

The frontend is a React app that uses @blueskyproject/finch components. Every service publishes its API as an OpenAPI schema (shared-schema/*.openapi.json) that the frontend turns into TypeScript types — so a route change on the backend shows up as a compile-time error in the UI. See Frontend integration for how the generation, WebSocket connections, and dev proxy work.

The target-state pivot: HTTP-only

What's changing

Historically queueserver_service spoke two protocols to the outside world: HTTP (for the browser) and 0MQ (for CLI tools and older clients). The target state after the current remediation is HTTP-only. Every 0MQ knob, environment variable, and library dependency gets removed; every consumer moves to HTTP.

The one remaining "internal" 0MQ use — the manager publishing console output — is replaced by an in-process asyncio queue. The upshot: fewer moving parts, one API surface to test, one contract to freeze.

See Key updates for the milestones that got us there (dispatch-serialization lock, the compat test suite, the staged 0MQ retirement, the HTTP-based qserver CLI, etc.).
