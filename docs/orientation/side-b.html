Side B · configuration ↔ queueserver

The conversation between the registry and the plan runner. Everything about "which devices does this plan need, and can I have exclusive access to them for the duration?" travels this seam.

The picture

1 · Device bootstrap + diff

When a queueserver worker opens an environment, its Python namespace starts with whatever the profile collection defines. But the beamline registry is the authoritative source — devices may have been added, removed, or reconfigured since the profile was written. Queueserver reconciles the two at three points:

What changed here

  • The bootstrap used to gate "registry empty?" on active-only device listings. If every device was disabled, bootstrap would re-enable them. Now the check uses the full registry and correctly treats "all disabled" as "no change needed".
  • The overlay used to be applied one device at a time with no rollback. It's now stage-then-commit inside the worker: build all of the new device instances in a staging dict; only mutate the namespace if every one succeeds.
  • The overlay RPC used to use the 0.5 s default pipe timeout — while the worker did a full list regeneration that took longer. A registry change during a running plan would time out and abort the plan. Now uses the long timeout.
  • The periodic sync task and the manager-restart path now run inside a guard that surfaces failures in /api/status (config_service_sync_error) instead of dying silently or crash-looping.

2 · Per-plan device locks (with optional lease/heartbeat)

The lock story has two parts that are easy to conflate. The lock lifecycle — acquire, release, force-unlock, epoch-detect — is always on. The lease/heartbeat mechanism (the "locks that expire if you don't renew them" story) is a distinct, opt-in feature layered on top.

Leases are opt-in — mind the default

Enabling leases sets CONFIG_LOCK_LEASE_TTL_SECONDS to a positive number; the default is 0, which disables expiry. With leases off, a lock is held until an explicit unlock or an admin force-unlock. If the queueserver crashes without releasing, the lock stays until the operator intervenes — or until configuration_service itself restarts (which clears the in-memory lock table). This is the historical behaviour, kept as default for compatibility.

The lock-authority epoch, by contrast, is always on. It's the mechanism that guarantees the reader (direct_control) and the writer (queueserver) can both detect a lock-table reset, regardless of whether leases are enabled.

Two dimensions of lock policy, four combinations

The behaviour operators actually see depends on two orthogonal knobs — one on each side of the seam:

queueserver lock_scope configuration lock_all What happens
Default plan False Only the devices the plan actually names are locked, and only while the plan is running. Idle env → nothing locked.
"Lock the beamline" plan True While a plan is running, every registered device reports as locked to direct_control — a plan holds the whole beamline. Toggle at runtime with PUT /lock/policy.
Legacy "environment" environment False Devices are locked for the lifetime of the worker environment — locked even between plans. Kept for backward compatibility.
Legacy + lock-all environment True Whole beamline locked whenever the environment is open.

What changed here

  • Locks used to be indefinite and in-memory only, with no expiry. An opt-in TTL lease + /lock/renew heartbeat now bounds the blast radius of a crashed lock holder. Off by default; enable with CONFIG_LOCK_LEASE_TTL_SECONDS > 0.
  • Queueserver's ConfigServiceCoordinator renews on a timer when the TTL is enabled, and re-acquires locks on lost_devices or a changed lock_epoch — closing the "config-service restart mid-plan drops locks" window.
  • Direct_control observes lock_epoch and logs lock_authority_reset when it changes, so a brief "everything available" window after a restart is attributable, not silent.
  • Queueserver's lock_scope default changed from environment to plan, so an idle environment no longer holds locks.
  • Per-plan locking now unions the devices named by the plan with the plan's bluesky decorator defaults, so plans that use decorator-attached devices don't hit "unowned device" errors.
  • Force-unlock is all-or-nothing (validate every device exists before releasing any), and always writes an audit row.

3 · Two epochs, one word — don't confuse them

The word "epoch" appears in two distinct mechanisms in the config↔queueserver conversation. Being clear about which is which saves debugging time:

service_epochlock_epoch
Also known asregistry epochlock-authority epoch
Lives onthe DB-backed device registrythe in-memory lock table
Changes whenthe registry is reset (or migrated)the configuration_service process starts
What it signalsyour cached device list is stale — re-fetchthe lock table was wiped — re-acquire
Who caresqueueserver's device-sync pathqueueserver's lock holder + direct_control's status reader

A plain configuration_service restart does change lock_epoch (locks are in-memory, gone on restart) but does not change service_epoch (the DB survived). This is exactly why both signals are needed — one on its own would miss half the failure modes.

How the pieces fit together — a full plan run

The mental model

Two rules

  1. Queueserver owns the plan; configuration owns the lock. The plan says "I need devices A, B, C"; configuration decides whether they're free and records who's holding them.
  2. Two epochs, two distinct signals. service_epoch tells you the registry moved; lock_epoch tells you the lock table was wiped. Optional TTL leases add a third safety net — self-releasing locks when the holder crashes — but they're off by default.
