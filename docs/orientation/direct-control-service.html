direct_control_service

The live wire to the hardware — where ophyd lives in the ophyd-service backend. Instantiates ophyd/ophyd-async device classes on demand, drives them, and streams updates back over WebSockets.

Why this is the ophyd service

"ophyd-service" is a project-level framing rather than a term the code uses, but direct_control is where the framing becomes concrete. It's the only service in the backend that instantiates ophyd device classes and drives their methods — everything else works with metadata, PV strings, and lock records. When a request arrives like "move motor X to 3.14 and wait for it to arrive", direct_control imports the ophyd class from the registry, calls motor.set(3.14), awaits the ophyd Status, and returns the result. That's ophyd — exposed as a service.

The service supports two device frameworks side by side, and it says so unambiguously — from src/direct_control/drivers.py:

"direct_control drives BOTH classic ophyd (pyepics transport) and ophyd-async (aioca/p4p transports) devices during the multi-year migration between the two frameworks. The two frameworks do not interoperate below the bluesky RunEngine: different transports, connection models, and Status semantics. This module is the single place that branches on framework — everything above it stays framework-agnostic."

Two drivers, one API surface

Both drivers expose the same three methods — connect(device, timeout), invoke(target, method, args, kwargs, timeout, use_put), destroy(device) — so the endpoint code above them stays framework-agnostic. The differences live inside the drivers:

ClassicOphydDriver → pyepics

Handles classes inheriting from ophyd.ophydobj.OphydObject. Because the underlying calls are blocking, every framework call is offloaded with asyncio.to_thread — wait_for_connection, the verb invocation, and Status.wait. This is the only way to bridge sync ophyd into the async web server without stalling the event loop.

Retained because a large body of existing device classes still uses classic ophyd. Fully supported.

OphydAsyncDriver → aioca + p4p

Handles classes inheriting from ophyd_async.core.Device. Everything is awaited natively — await device.connect(), await AsyncStatus, no threadpool hop. The driver even hard-refuses a classic Status returned by an async-tagged class ("framework mix-up") rather than deadlock on the wrong event loop.

The direction of travel. ophyd-async is what the code paths were built for.

How the dispatch actually works

Verified from drivers.py:

  1. A device class arrives from the registry. detect_framework(cls) checks issubclass(cls, AsyncDevice) first, then issubclass(cls, OphydObject).
  2. A class belonging to neither is a hard error — there is no PV-gateway fallback at the device-control level. The docstring is explicit: "cannot drive it".
  3. The right driver instance is returned by driver_for(framework), and every subsequent call for that device dispatches through it.
  4. Method names are also allowlisted (set, put, get, read, describe, read_configuration, describe_configuration, trigger, stop) — the bluesky verb surface, not arbitrary attribute dispatch. A control endpoint that executes any named method on live hardware would be an injection surface.

Dependencies confirm the story

From backend/direct_control_service/pyproject.toml, with its own explanatory comments:

# Bluesky ecosystem. ophyd brings the pyepics CA transport for classic
# devices; ophyd-async[ca] brings aioca for async devices.
"ophyd>=1.9.0",
"ophyd-async[ca]>=0.13.0",

# EPICS integration
"pyepics>=3.5.0",
"p4p>=4.1.0",

Both transports are declared as direct dependencies, both device frameworks pulled in with their EPICS extras. ophyd-async[ca]'s [ca] extra is what brings in aioca; p4p gives the PVAccess side.

PV path vs device path — the README's own labels

The service's own README calls the two paths by canonical names:

  • Low-fidelity channel · PV Control — raw EPICS set/get by PV name. Fast, cheap, still coordination-gated. The survivor of the pyepics-wrapper era.
  • High-fidelity channel · Device Method Execution — "instantiates the registry's device class as a live ophyd object and runs the verb with Status completion awaited. Supports both classic ophyd and ophyd-async device classes, dispatched per device by framework detection."

The high-fidelity channel is the one that makes this an ophyd service. The low-fidelity channel is the useful fallback — because sometimes you just want to poke a PV without materializing a device object.

The picture

What it does (in plain English)

Reads and writes PVs

The lowest-level path: given a PV name and a value, do the EPICS put. Values, timestamps, and errors are surfaced verbatim. Optionally validates against control limits before writing.

Runs device methods

The higher-level path: given a device name and a method (set, trigger, stop, …), instantiate the ophyd class from the registry, call the method, and await completion. Works for classic ophyd and ophyd-async devices.

Streams live values

Frontend widgets open a WebSocket, subscribe to PVs or devices, and get a push every time a value changes. Alarm severity comes with every update.

Never writes without permission

Every write path first asks configuration_service "is this device locked?". If it is, direct_control returns 423 Locked and never touches EPICS.

Key API surfaces

REST endpoints

RouteWhat it's forNotes
GET /api/v1/pv/{name}/value Read one PV right now. No lock check (read is free).
POST /api/v1/pv/set Write one PV. Coordination-gated. 423 if locked.
POST /api/v1/pv/set-batch Write many PVs in one round-trip. Per-item results; partial failures reported.
POST /api/v1/devices/{name}/{method} Invoke a method on the ophyd device (e.g. motor/set). Instantiates the class; awaits Status.
GET /api/v1/devices/{name}/components Walk the ophyd device hierarchy (for the tree view). Nested-device navigation.
POST /api/v1/devices/enrich Given a list of device specs, connect them and report metadata. Called by configuration during registry builds.

WebSocket streams

EndpointPurpose
/ws/pv-socketSubscribe to a set of PV names; get value + metadata + alarm on every change.
/ws/device-socketSubscribe to a device; get updates for every PV it owns, keyed by the device name.
/ws/control-socketCommand channel over WS — write PVs and receive per-write acks (also coordination-gated).

The safety story

Live monitoring: one subscription, many clients

Target-state improvements

Blocking EPICS work out of the hot lock

The EPICS connect (up to 5 s for a dead PV) used to happen while holding the global monitor lock — a single dead PV could freeze every other subscriber. Connects now happen outside the lock behind a per-PV connect lock. Subscribes to a batch of PVs happen concurrently, not serially.

delivered

No more monitor leaks

If a client disconnected mid-subscribe, the EPICS monitor could stick around forever. After the subscribe returns we now re-check whether any client still wants the PV — if not, we unsubscribe immediately. Both the pv-socket and device-socket got the fix.

delivered

Alarm severity is real data now

Every PV update carries the EPICS (status, severity) pair and a friendly name (NO_ALARM, MINOR, MAJOR, INVALID). Widgets can highlight alarming values without a separate query.

delivered

read_only actually means read-only

subscribeReadOnly used to be cosmetic — a writable signal was still built. It now plumbs all the way down to the ophyd signal, so an accidental write on a read-only subscription fails at the layer that should catch it.

delivered

Failed subscribes tell you why

Instead of silently going quiet, a subscribe that fails now sends a pv_subscribe_failed error envelope over the socket. The subscribed confirmation is only sent for PVs that actually stuck.

delivered

Coordination unknown ≠ hardware failure

When the coordination check returns an unmodelled status, we now return 409 (a coordination problem) instead of 500 (a hardware problem). Crucially we don't report PV health on it — a gate failure was never about the PV.

delivered

Stop-on-timeout for commands

If a motor.set(x) times out, we now call motor.stop() — hardware shouldn't be left moving because we gave up waiting. Opt out per-request or with stop_on_command_timeout=false.

delivered

Device socket works in standalone mode

The device WebSocket used to bypass the registry abstraction and only work with a live configuration_service. It now goes through the same RegistryProvider as the write path, so file-mode and standalone deployments work end-to-end.

delivered

Control-limit validation on writes

Numeric writes now read the PV's (lower_ctrl_limit, upper_ctrl_limit) and reject out-of-range values with 422 before touching EPICS. On by default; opt out per-request.

delivered

Update fan-out coalescing (planned)

High-rate PVs can spam clients faster than they can render. A per-PV latest-value coalescing policy will drop stale updates when a client is slow, and TTL-evict monitors created by REST reads without follow-up subscribers.

planned

Rule of thumb

direct_control_service never makes decisions about who is allowed to write or whether a plan is running. Both of those live in configuration_service. Direct_control just executes what it's told, once it has clearance.
